## 1. Product Positioning

### 1.1 Elevator Pitch

Memory Governance Console is the visual management interface for opencode-memory — an AI agent memory infrastructure platform. It provides observability, governance, and human-in-the-loop approval for autonomous memory management.

### 1.2 Design References

| Product | Why |
|---------|-----|
| LangSmith | LLM Observability — trace, evaluate, debug |
| Kubernetes Dashboard | System governance — cluster-level health, resource management |
| GitHub PR | Peer review workflow — diff, approve, reject |
| Linear | Product experience — clean, fast, high information density |
| Neo4j Browser | Relationship exploration — graph visualization |

### 1.3 Core Principles

- **Explainable AI**: Every memory operation shows why, what evidence, what risk
- **Human approval**: Governance actions require explicit human sign-off (opt-in auto-execute)
- **High information density**: Developer tool feel — show more, scroll less
- **Dark mode first**: AI infrastructure default
- **Provenance-first**: Every memory card shows source, confidence, lifecycle

## 3. Backend Architecture

### 3.1 HTTP Server

Plugin-embedded Hono server, same lifecycle as the plugin process.

| Aspect | Decision |
|--------|----------|
| Framework | **Hono** (4KB, zero dependencies, Bun native) |
| Transport | HTTP + WebSocket (SSE fallback) |
| Port | Dynamic, randomized on startup |
| Discovery | `memoryDir/http.json` — `{ port, pid }` |

### 3.2 API Endpoints

```
Phase 1 (core):
  GET  /api/v1/health              → state.json health + pipeline status
  GET  /api/v1/memories            → paginated list, filter (scope/type/status), search
  GET  /api/v1/memories/:id        → single memory detail + provenance
  GET  /api/v1/memories/:id/history → lifecycle events (create/update/merge/archive/restore)

Phase 2 (governance):
  GET  /api/v1/governance          → scheduler status (last run, next run, config)
  GET  /api/v1/governance/proposals → pending proposals (paginated, sortable)
  POST /api/v1/governance/proposals/:id/approve → approve proposal
  POST /api/v1/governance/proposals/:id/reject  → reject proposal with reason
  GET  /api/v1/audit               → audit log (filterable by action/actor/risk/result)

Phase 3 (advanced):
  GET  /api/v1/conflicts           → conflict clusters
  GET  /api/v1/pipeline            → hook pipeline status per stage
  GET  /api/v1/evaluation          → benchmark results & quality scores
  WS   /api/v1/events              → real-time event stream

Health check:
  GET  /api/v1/health             → { status, uptime, memoryCount, pipelineStatus }
```

### 3.3 Data Flow

```
Memory files (.md) + state.json
        │
        ▼
  Hono API server (in-plugin)
        │
        ▼
  Next.js 15 SPA (static)
        │
        ▼
  shadcn/ui components
```

## 6. Frontend Technology Stack

| Layer | Choice | Rationale |
|-------|--------|-----------|
| Framework | Next.js 15 (App Router) | Static export, SSR optional |
| Language | TypeScript 5.x | Strict mode |
| Styling | Tailwind CSS v4 | Utility-first, design tokens |
| Components | shadcn/ui (customized) | Copy-source, fully customizable |
| State | TanStack Query | Server state, caching, polling |
| Global State | Zustand | Lightweight, no boilerplate |
| Graphs | React Flow | Custom nodes, edges, minimap |
| Charts | Recharts | Composable, React-native |
| Animation | Framer Motion | Micro-interactions, layout animations |
| Diff | Monaco Editor (diff mode) | For approval center diffs |

## 9. Error Handling & Edge Cases

| Scenario | Behavior |
|----------|----------|
| Server unreachable | Dashboard shows "Connection lost" banner, retry every 5s |
| Empty memory store | Memory Explorer shows empty state with "No memories yet" + CTA |
| Proposal approval fails | Toast error, proposal stays in pending list |
| WebSocket disconnected | Auto-reconnect with exponential backoff (1s, 2s, 4s, max 30s) |
| Large memory list (>1000) | Server-side pagination (default 50/page), virtual scrolling |
| Concurrent approval | Lock check on server, "already approved" message on conflict |
| State file corrupt | API returns degraded status, UI shows warning banner |

## 11. Design Constraints

### 11.1 What NOT to build

- **No CRUD admin panel**: This is not a generic database viewer. Skip bulk edit, import/export CSV, generic table views.
- **No login/auth**: Runs on localhost, plugin-embedded. No user management.
- **No real-time collaboration**: Single-user, single-instance.
- **No mobile responsive**: Desktop-first, minimum 1280px width.

### 11.2 What to AVOID (anti-patterns)

- Bootstrap cards layout
- Crowded data tables with no hierarchy
- Generic "admin dashboard" with identical card widgets
- Toast notifications for every action (use activity feed instead)
- Auto-refresh without user awareness (show a "Live" indicator)
