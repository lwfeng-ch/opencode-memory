# v0.5.0 Discovery Engine — Spec

> **Status**: Draft  
> **Date**: 2026-07-23  
> **Base**: v0.4.3 — Memory Governance Foundation (1006 tests, 97 files)  
> **Target**: 1150+ tests, 0 regression

---

## 1. Motivation

v0.4.x 建立了 Memory Governance Foundation：provenance（可信度）、worthiness（价值）、conflict（冲突）、recall feedback（召回反馈）。系统能回答"每条记忆是否值得存在"。

但系统仍然不知道：

- 哪些记忆属于同一主题？
- 哪些记忆描述同一事实但说法不同？
- 哪些记忆可以合并、压缩或归档？

v0.5.0 Discovery Engine 解决这个问题：**从治理信号中发现可合并、可压缩、可修复的记忆集合，并生成 evidence-based proposal**。

---

## 2. 设计原则

| 原则 | 含义 |
|------|------|
| **Discovery only, no execution** | v0.5.0 只发现和提案，不执行任何修改 |
| **Evidence-based, not LLM-driven** | 所有决策基于规则和治理信号，LLM 只负责解释和摘要 |
| **Reuse before build** | 优先复用 v0.4.x 现有组件（ConflictGraph、fingerprint.ts、WorthinessScore） |
| **Backward compatible** | 旧 `src/dream.ts` 保留，通过 `DreamMode` 切换 |
| **Deterministic core** | Cluster 和 Candidate 生成是纯函数，相同输入产生相同输出 |

## 5. MemoryCluster 系统

### 5.1 ClusterLevel

```typescript
// src/dream/cluster/types.ts

export type ClusterLevel = "topic" | "claim" | "conflict"

export interface MemoryCluster {
  id: string
  level: ClusterLevel
  members: string[]          // 文件名列表
  confidence: number         // 0-1
  metadata: {
    topic?: string           // topic 级别才有
    subject?: string         // claim 级别才有
    predicate?: string       // claim 级别才有
    values?: string[]        // claim 级别才有
    conflictIds?: string[]   // conflict 级别才有
  }
}
```

### 5.2 三级 Cluster 关系

```
Topic (宽泛): 同一主题, 关键词相似
  |
  | contains
  |
Claim (精确): 同一 subject/predicate, 不同值
  |
  | conflict
  |
Conflict (冲突): ConflictGraph 连通分量
```

### 5.3 TopicCluster

```typescript
// src/dream/cluster/topic-cluster.ts

export class TopicClusterBuilder {
  constructor(
    private fingerprint: { extractKeywords: (text: string) => Set<string> }
  )

  /** 对所有 active memory 文件做 pairwise Jaccard 相似度计算，
   *  相似度 >= threshold 的归入同一 topic cluster。 */
  build(files: string[], threshold: number = 0.4): MemoryCluster[]
}
```

实现：

- 复用 `src/evaluation/fingerprint.ts` 的 `extractKeywords` 和 `jaccardSimilarity`
- Pairwise 比较，O(n²) 但 n 通常在 100-500 范围，可接受
- 可选的 `maxClusterSize` 限制（默认 20，防止超大 cluster）
- 单文件 cluster 不输出（至少 2 个成员）
- **Note**: threshold 在 v0.5.0 写死为 0.4，v0.5.1 引入配置化

### 5.4 ClaimCluster

```typescript
// src/dream/cluster/claim-cluster.ts

export class ClaimClusterBuilder {
  constructor(
    private claimExtractor: ClaimExtractor
  )

  /** 对所有 memory 提取 claim，按 subject+predicate 分组，
   *  同一组内 values 不同则形成 ClaimCluster。 */
  build(memories: MemoryFile[]): MemoryCluster[]
}
```

实现：

- 复用 `src/conflict/claim-extractor.ts` 的 `ClaimExtractor`
- 按 `(subject, predicate)` 分组，不要求 values 冲突（兼容即可）
- 单值 cluster 不输出（至少 2 个不同值）

### 5.5 ConflictClusterAdapter

```typescript
// src/dream/cluster/conflict-adapter.ts

export class ConflictClusterAdapter {
  /** 从 ConflictGraph 的连通分量直接生成 ConflictCluster。
   *  不重新检测冲突，只做格式转换。 */
  fromGraph(graph: ConflictGraph): MemoryCluster[]
}
```

实现：

- 输入：`ConflictGraph`（来自 `src/conflict/graph.ts`）
- 每个连通分量 → 一个 `MemoryCluster`（level="conflict"）
- 保留 `conflictIds` 追踪原始冲突边
- 空图返回空数组

### 5.6 ClusterBuilder（编排器）

```typescript
// src/dream/cluster/builder.ts

export class ClusterBuilder {
  constructor(
    private topicBuilder: TopicClusterBuilder,
    private claimBuilder: ClaimClusterBuilder,
    private conflictAdapter: ConflictClusterAdapter,
  )

  /** 按顺序执行三级聚类：
   *   1. topic — 宽泛主题分组
   *   2. claim — 在 topic 内部进一步精确匹配
   *   3. conflict — 从 ConflictGraph 取连通分量
   *  返回所有 cluster 的合并列表。 */
  buildAll(): MemoryCluster[]
}
```

## 6. Evidence Analyzer

### 6.1 Evidence 收集

```typescript
// src/dream/analyzer/evidence.ts

export interface EvidenceReport {
  clusterId: string
  members: Array<{
    filename: string
    worthiness: number
    confidence: number
    usage: number
    feedback: number
    temporal: number
  }>
  /** 如果存在一个 memory 在所有维度上显著优于其他，标记为 dominant */
  dominant?: string
}

export class EvidenceCollector {
  collect(cluster: MemoryCluster, memories: MemoryFile[]): EvidenceReport
}
```

### 6.2 Cluster Score

```typescript
// src/dream/analyzer/cluster-score.ts

export class ClusterScoreEngine {
  /** 对 cluster 内的 pairwise 关系评分：
   *  - 兼容性：相同 claim 且 values 不冲突
   *  - 主导力：是否存在 dominant memory
   *  - 冗余度：pairwise 相似度分布 */
  score(cluster: MemoryCluster, evidence: EvidenceReport): {
    compatibility: number
    dominance: number
    redundancy: number
  }
}
```

### 6.3 LLM Explain Layer

```typescript
// src/dream/analyzer/explain.ts

export class ExplanationGenerator {
  /** 生成人类可读的合并理由。
   *  LLM 只负责解释，不参与决策。
   *  输入：已确定的 Cluster + Evidence
   *  输出：自然语言解释字符串 */
  generate(cluster: MemoryCluster, evidence: ClusterEvidence): string
}
```

**LLM 限制**：

| 允许 | 禁止 |
|------|------|
| 总结 cluster 的主题 | 判断哪个 memory 更好 |
| 解释为什么两个 memory 相似 | 决定是否合并 |
| 生成合并后的预览文本 | 选择保留哪个、删除哪个 |
| 描述冲突的内容差异 | 修改原始 memory 内容 |

## 7. ConsolidationCandidate

```typescript
// src/dream/candidate/types.ts

export type CandidateAction =
  | "merge"
  | "archive"
  | "split"
  | "keep"

export interface ConsolidationCandidate {
  id: string
  clusterId: string
  action: CandidateAction
  targets: string[]            // 文件名列表
  evidence: EvidenceReport     // 复用 analyzer 的 evidence
  expectedGain: number         // 0-1, 预期治理收益
  riskHint: number             // 0-1, 风险提示（供 v0.5.1 Decision Engine 使用）
  confidence: number           // 0-1
}
```

### 候选生成规则

| 条件 | 建议操作 |
|------|---------|
| 同一 topic, 高相似度, 无冲突, 兼容 claim | `merge` |
| 同一 topic, 低 worthiness, 无 recall, 被 superseded | `archive` |
| 同一 topic, 低 worthiness, 无 recall, 未被 superseded | `archive` |
| 同一 topic, 冲突 claim, 不可兼容 | `split`（标记冲突） |
| 同一 topic, 无重叠, 低 worthiness, 无冲突 | `keep`（不生成 candidate） |

## 8. Candidate Generator

```typescript
// src/dream/candidate/generator.ts

export class CandidateGenerator {
  constructor(
    private evidenceCollector: EvidenceCollector,
    private clusterScore: ClusterScoreEngine,
  )

  /** 对每个 cluster：
   *   1. 收集 evidence
   *   2. 执行 cluster score（兼容性 + 主导力 + 冗余度）
   *   3. 应用规则引擎 → 生成 ConsolidationCandidate
   *   4. 按 confidence 降序排列 */
  generate(clusters: MemoryCluster[], memories: MemoryFile[]): ConsolidationCandidate[]
}
```

```typescript
// src/dream/candidate/scorer.ts

export class CandidateScorer {
  /** 计算 candidate 的 confidence 和 expectedGain。
   *  基于 evidence 中的 worthiness/usage/feedback/temporal 综合评分。 */
  score(candidate: ConsolidationCandidate): {
    confidence: number
    expectedGain: number
    riskHint: number
  }
}
```

**规则引擎**（非 LLM）：

```typescript
function evaluateAction(
  cluster: MemoryCluster,
  evidence: EvidenceReport,
  clusterScores: { compatibility: number; dominance: number; redundancy: number }
): CandidateAction
```

规则：

1. 如果 cluster 内所有 memory 属同一 topic 且高相似度且无冲突 → `merge`
2. 如果 cluster 内存在 dominant memory 且其他 memory 低 worthiness → `supersede`
3. 如果 cluster 内所有 memory 低 worthiness (<0.2) 且无 recall → `archive`
4. 如果 cluster 内高相似度且同 content → `deduplicate`
5. **Note**: 以上阈值 v0.5.0 写死，v0.5.1 移入可配置的 decision policy

## 9. Proposal Mapper

```typescript
// src/dream/proposal.ts

export class ProposalMapper {
  /** 将 ConsolidationCandidate 转换为 RepairCandidate。
   *  RepairCandidate 是 v0.3.3 RepairQueue 的输入格式。 */
  toRepairCandidate(candidate: ConsolidationCandidate): RepairCandidate
}
```

**映射规则**：

```
ConsolidationCandidate         RepairCandidate
─────────────────────          ───────────────
action: "merge"        →      type: "merge", action: "archive" (keep winner)
action: "archive"      →      type: "lifecycle_archive", action: "archive"
action: "split"        →      type: "lifecycle_archive" (no split in RepairCandidate)
action: "keep"         →      (不生成)
```

注意：`merge` 在现有 `RepairCandidate` 中没有直接对应。v0.5.0 的 `merge` 映射为"保留高 worthiness 的一方，archive 另一方"。

## 14. 增量 vs 完整

v0.5.0 设计为**增量实现**，按架构验证价值排序：

```
Phase 0: 基础类型定义（types / mode）
Phase 1: ConflictAdapter（验证 v0.4.2 → v0.5.0 数据契约）
Phase 2: ClaimCluster（精确语义聚类）
Phase 3: TopicCluster（模糊语义聚类，引入 fingerprint.ts）
Phase 4: DreamRunner + Evidence + Candidate（完整 pipeline）
Phase 5: ProposalMapper + RepairQueue 集成
```

每个 Phase 独立可测试，不阻塞后续 Phase。Phase 1 优先验证架构闭环，而非聚类算法。
